# Kadenz Scanner

[![CI](https://github.com/kadenz-live/kadenz-scanner/actions/workflows/ci.yml/badge.svg)](https://github.com/kadenz-live/kadenz-scanner/actions/workflows/ci.yml)

[![License: MIT](https://img.shields.io/badge/License-MIT-informational.svg)](LICENSE)

Cross-platform iOS + Android app for door staff to validate event tickets by
scanning their QR code — online against the Kadenz API, or fully offline from a
pre-downloaded, signed manifest.

This is the ticket-validation client for [Kadenz](https://kadenz.live). It is
open source on purpose: **you can verify exactly how tickets are checked.** The
Kadenz backend that issues and signs tickets is a separate, proprietary service.

## Why the offline validation is trustworthy

The scanner can admit ticket-holders with **no network connection** — at a
festival gate, in a basement venue — and it does so *without ever holding a
signing secret.*

- The server signs each ticket's QR token (HMAC) and publishes an **offline
  manifest**: a list of SHA-256 digests of the signed tokens, plus a collapsed
  status per ticket.
- Offline, the scanner hashes the scanned QR token (SHA-256) and looks that
  digest up in the manifest. A match means the server already signed this exact
  token; no match means reject.
- **The HMAC signing secret never reaches the device.** The scanner can
  therefore recognise valid tickets, but it can neither forge nor mint them.

See [`lib/models/offline_manifest.dart`](lib/models/offline_manifest.dart) and
[`lib/services/offline_validator.dart`](lib/services/offline_validator.dart) —
that is the whole trust story, in the open.

## Why are `ios/` and `android/` not in the repo?

The native platform projects (Xcode workspace, Gradle wrapper, default app
icons, `.pbxproj` files) are **generated by the Flutter SDK** so they match your
installed Flutter version exactly. Shipping them would mean committing binary
blobs and version-pinned project files that break across Flutter releases.

The Dart source — the actual app logic — lives in `lib/`; the native scaffolding
is regenerated on demand by `setup.sh`.

## First-time setup

```bash
./setup.sh
```

The script:

1. Runs `flutter create .` to materialise the iOS and Android projects.
2. Runs `flutter pub get`.
3. Overlays project-specific patches from `_native_overrides/` — camera
   permission in `Info.plist` / `AndroidManifest.xml`, plus a relaxed ATS policy
   so the app can reach a local HTTP API during development.

Run once after cloning. To change the bundle org id, run
`ORG_ID=com.yourco.kadenz ./setup.sh` (defaults to `com.example`).

## Run

```bash
flutter run                       # production API (https://kadenz.live)
flutter run -d "iPhone 15 Pro"
flutter run -d emulator-5554
```

### API base URL

The scanner resolves its API base URL in this order:

1. **Compile-time** — `--dart-define=KADENZ_API=<url>`. Wins over everything.
2. **Stored override** — set via the in-app settings screen (gear icon),
   persisted to `flutter_secure_storage`.
3. **Default** — `https://kadenz.live` (compiled in).

The settings screen surfaces the active URL and its source. The login screen
shows a loud amber banner if the resolved URL is not HTTPS or if `GET /healthz`
does not return 2xx.

#### Local development overrides

```bash
# iOS simulator
flutter run --dart-define=KADENZ_API=http://localhost:3000

# Android emulator (uses host loopback alias 10.0.2.2)
flutter run --dart-define=KADENZ_API=http://10.0.2.2:3000

# Physical device on the same Wi-Fi
flutter run --dart-define=KADENZ_API=http://<MAC-IP>:3000
```

iOS allows `http://` to localhost / `*.local` / RFC1918 via the
`NSAllowsLocalNetworking` ATS exception in
`_native_overrides/ios/Info.plist.patch.plist`. Production traffic still must be
HTTPS — that ATS key does **not** widen for public hosts.

To sign in you need a `scanner`- or `admin`-role account on the Kadenz backend
you point the app at. A local development backend seeds its own demo accounts.

## Build for release

```bash
flutter build apk --release   --dart-define=KADENZ_API=https://kadenz.live
flutter build ios --release   --dart-define=KADENZ_API=https://kadenz.live
```

CI (`.github/workflows/release.yml`) pins `--dart-define=KADENZ_API=https://kadenz.live`
explicitly so the production endpoint is part of the release record.

## Features

- **Login** as a `scanner` or `admin` user against the Kadenz API.
- **Event picker** lists today's and upcoming events.
- **Camera scanner** (`mobile_scanner`) reads ticket QR payloads.
- **Live validation** against `POST /api/v1/scanner/tickets/validate`.
- **Offline validation** from a signed, pre-downloaded manifest (see above).
- **Visual + audio feedback** with green / amber / red result panels.
- **App Attest** device attestation before the backend trusts a scanner client.
- **Configurable API endpoint** persisted via `flutter_secure_storage`.
